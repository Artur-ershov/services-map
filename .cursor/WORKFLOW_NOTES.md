# Заметки о процессе работы

## Подход к документации

**Принцип 80/20:**
- 80% кода - без документации (очевидно)
- 20% кода - с комментариями (важные решения)

**Правило "3 вопроса":**
Документировать, если хотя бы один ответ "да":
1. Понятно ли это через 3 месяца?
2. Можно ли сделать по-другому?
3. Влияет ли это на другие части?

**Правило "Why, not What":**
- ❌ НЕ документировать что делает код (читается из кода)
- ✅ Документировать почему так сделано (неочевидные решения)

---

## Процесс работы

**Ежедневно:**
1. AI автоматически добавляет комментарии-метки при написании кода
2. Только важные решения получают объяснение "почему"
3. Временные решения - с датой удаления

**При принятии решений:**
- Использовать шаблон комментариев: `@ARCH`, `@TEMP`, `@CRITICAL`
- Объяснять "почему", не "что"
- Минимум документации - максимум понимания

**Восстановление контекста:**
- Читать ARCHITECTURE_DISCUSSIONS.md в начале сессии
- Использовать комментарии-метки в коде
- Обращаться к README.md для критичной информации

---

## Метрика перегрузки

Если документирование занимает >20% времени → упростить
Если код не читается без документации → упростить код
Если вопрос повторяется >3 раз → задокументировать ответ

---

## Современные подходы к работе с AI

**Принцип минимализма:**
- Не создавать "систему поддержки системы"
- Документировать только критичное
- Использовать комментарии в коде как основную документацию

**Эффективные практики:**
1. **Structured comments** - комментарии-метки (`@ARCH`, `@TEMP`, `@CRITICAL`)
2. **Context files** - отдельные файлы для обсуждений (этот репозиторий)
3. **Incremental documentation** - документировать только по необходимости
4. **Code-first** - код как документация, текст - для объяснений "почему"

**Что НЕ работает:**
- Избыточная документация вокруг каждого куска кода
- Отдельные документы для очевидных вещей
- Документация "что делает" вместо "почему так"

**Что работает:**
- Комментарии в коде для важных решений
- Отдельные файлы для архитектурных обсуждений (internal)
- README только с критичной информацией

## Структура проекта

```
.cursor/                    # Внутренний процесс (gitignored)
  ├── ARCHITECTURE_DISCUSSIONS.md  # Обсуждения и идеи
  └── WORKFLOW_NOTES.md            # Заметки о процессе

README.md                   # Только критичное для проекта
CODE_COMMENTS.md            # Если нужно, но лучше в коде
```

**Когда добавлять в .cursor/:**
- Внутренние обсуждения с AI
- Временные идеи и эксперименты
- Контекст для восстановления работы

**Когда НЕ добавлять:**
- Публичная документация → README.md
- Критичная информация → комментарии в коде
- Очевидные вещи → не документировать
